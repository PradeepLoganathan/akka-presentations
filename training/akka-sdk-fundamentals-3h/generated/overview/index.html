Positioning

Not a library. A runtime with an SDK.

You write
  • Components: entities, views, endpoints, workflows, timers, consumers, agents
  • Plain Java records & classes with a few annotations
  • Maven module — nothing exotic
Akka runs
  • Persistence, sharding, cluster membership
  • Message delivery guarantees & back-pressure
  • HTTP surface, ACLs, observability
  • Rolling deploys, autoscale, multi-region
The SDK's job is to keep distribution invisible while you write domain logic.

Event sourcing

Two ways to remember.

CRUD / row overwrite
  • Table row holds the current state
  • UPDATE erases what was there
  • History exists only if you built it (audit tables, triggers)
  • Answers "what is X now?"
Event sourced
  • Append-only log of things that happened
  • Current state = fold over past events
  • History is the storage model, not an add-on
  • Answers "what is X now, and how did it get there?"
Your bank shows you a statement (events), not just a balance (state).
Event sourcing

What the log buys you.

Once the source of truth is the event stream, several capabilities fall out for free.

Audit

Every change is a durable fact. Compliance stories write themselves.

Replay

Reconstruct any past state by folding events up to a point in time.

New projections

Ship a new read model tomorrow by replaying yesterday's events into it.

Debugging

Bug reports say "at 14:03". You have the event that fired at 14:03.

Domain fit

Business folks already speak in events: "shipped", "cancelled", "paid".

Integration

Events are a natural fan-out to downstream systems & analytics.

Event sourcing

Great tool. Not the only tool.

Reach for it when
  • History matters — audit, compliance, disputes
  • Multiple read models over the same domain
  • "What did X look like on Tuesday?" is a real question
  • Domain has a natural event vocabulary
Skip it when
  • State is a settings page — latest is all you need
  • You have no domain narrative, just CRUD
  • Aggregate would grow an unbounded event stream you'll never trim
  • Team hasn't internalized eventual consistency yet
Rule of thumb: if who changed what, when, and why matters — event source it.
Commands & events

Commands are intent.

A command is a request to change state. The entity decides whether to accept it.

domain/commands/CustomerCommands.java
public final class CustomerCommands {

  public record CreateCustomer(
      String firstName,
      String lastName
  ) {}

  public record DeleteCustomer() {}
}
  • Named in the imperative — CreateCustomer, not CustomerCreated
  • Present tense: "do this"
  • Plain Java records: the field list is the contract
  • Can be rejected: "Customer already exists"
  • Never stored. They produce events, then vanish.
Commands & events

Events are facts.

An event describes something that already happened. Never rejected, never edited.

domain/events/CustomerEvents.java
public sealed interface CustomerEvents {

  record customerCreated(
      String customerId,
      String firstName,
      String lastName
  ) implements CustomerEvents {}

  record customerDeleted(
      String customerId,
      String firstName,
      String lastName
  ) implements CustomerEvents {}
}
  • Past tense — customerCreated
  • Immutable. Journaled forever.
  • sealed interface → the reducer's switch is exhaustive
  • Serialized to storage — treat schema changes with care
  • Downstream views & consumers subscribe to them

Entities

An entity is the actor model, tamed.

Each entity instance is an actor under the hood. The SDK hides sharding, mailbox, and lifecycle so you write plain Java.

Addressable

Reached by an ID string. customer-42 is one instance.

Single-writer

One instance handles its commands serially. No locking, no races.

Sharded

The runtime places instances across the cluster. You don't manage that.

Persistent

State (value) or events (event-sourced) are durable. Restarts don't lose data.

Passivated

Idle instances swap out of memory. Rehydrated on next command.

Two flavors

Value = latest snapshot. Event-sourced = event log.

Entities

Value entity: snapshot in place.

Simplest flavor. Handlers replace the whole state. No event log.

domain/entities/CustomerPreferencesEntity.java
@ComponentId("customer-preferences")
public class CustomerPreferencesEntity
    extends KeyValueEntity<CustomerPreferences> {

  @Override
  public CustomerPreferences emptyState() {
    return CustomerPreferences.defaults();
  }

  public Effect<String> set(SetPreferences cmd) {
    var next = new CustomerPreferences(
        cmd.theme(), cmd.locale(), cmd.marketingOptIn());
    return effects()
        .updateState(next)
        .thenReply(__ -> "OK");
  }
}
  • Extends KeyValueEntity<State>
  • No events, no reducer — just updateState(next)
  • Runtime persists the latest state atomically
  • Great for preferences, sessions, feature flags
  • Loses history — last write wins
Entities

Event-sourced entity: the log is the truth.

Command handlers persist events; state is rebuilt by folding them.

domain/entities/CustomerEntity.java
@ComponentId("Customer")
public class CustomerEntity
    extends EventSourcedEntity<CustomerState, CustomerEvents> {

  @Override
  public CustomerState emptyState() {
    return CustomerState.empty();
  }

  public Effect<String> create(CreateCustomer cmd) {
    if (currentState().customerId() != null)
      return effects().error("Customer already exists");

    var id = commandContext().entityId();
    var ev = new customerCreated(id, cmd.firstName(), cmd.lastName());
    return effects().persist(ev).thenReply(__ -> id);
  }
}
  • Extends EventSourcedEntity<State, Event>
  • @ComponentId = stable identifier used for storage
  • Validate first, then .persist(event)
  • commandContext().entityId() = which instance this is
  • State is not mutated here — the reducer does that
Entities

The Effects API: describe, don't do.

Handlers return an Effect. The runtime performs the persist, the reply, the error. This is what makes distribution invisible.

Effect builders
effects()
    .persist(event)
    .thenReply(state -> state.id());

effects()
    .persist(event1, event2)
    .thenReply(state -> "OK");

effects().error("Customer already exists");

effects().reply(currentState());
  • persist(…) — journal one or more events atomically
  • thenReply(state -> …) — runs after persistence, with new state
  • error(…) — reject the command, nothing persisted
  • reply(…) — query-only, no events
  • You never write to storage directly. Ever.
Entities

applyEvent — the reducer.

One method. One event in, one new state out. Called on every new event and on every replay.

CustomerEntity.applyEvent
@Override
public CustomerState applyEvent(CustomerEvents event) {
  return switch (event) {
    case customerCreated e ->
      new CustomerState(
          e.customerId(), e.firstName(), e.lastName(), false);
    case customerDeleted e ->
      new CustomerState(
          e.customerId(), e.firstName(), e.lastName(), true);
  };
}
  • Pure function. No I/O, no side effects.
  • Runs during command handling and during restart replay
  • Sealed events + switch expression = compiler-checked exhaustive
  • Return a new record — states are immutable
  • Adding an event? Adding a switch case is required by the compiler.
If applyEvent is not pure, replay after a restart will diverge from live behavior.
Entities

Which flavor for which problem?

Value entity
Storage modelLatest state snapshot
HistoryNot preserved
Best forPreferences, sessions, feature flags, caches
CostCheapest to write, cheapest to reason about
Event-sourced entity
Storage modelAppend-only event log
HistoryComplete, replayable

Views

Views are the read side.

Entities are optimized for writes and single-ID lookups. Views answer "list all customers" without hammering every entity instance.

Subscribes

To an entity's events (event-sourced) or state (value).

Projects

Each event updates rows in a queryable table.

Serves queries

@Query methods return rows filtered by criteria.

Multiple views can subscribe to the same events. Different shape, same source of truth — classic CQRS.
Views

CustomerView, annotated.

views/CustomerView.java
@ComponentId("customerview")
public class CustomerView extends View {

  public record customerEntry(
      String customerId, String firstName,
      String lastName, boolean deleted) {}

  @Query("SELECT * FROM customers_table")
  public QueryEffect<customerEntry> getAll() {
    return queryResult();
  }

  @Table("customers_table")
  @Consume.FromEventSourcedEntity(CustomerEntity.class)
  public static class CustomersUpdater
      extends TableUpdater<customerEntry> {

    public Effect<customerEntry> onEvent(customerCreated e) {
      return effects().updateRow(
          new customerEntry(e.customerId(),
              e.firstName(), e.lastName(), false));
    }
  }
}
  • @Table names the queryable table
  • @Consume.FromEventSourcedEntity wires the subscription
  • One onEvent(…) per event type
  • effects().updateRow(…) writes into the table
  • @Query is a SQL-like DSL — parameters bind to method args
  • No JPA, no repository — the runtime is your storage
Views

Eventual consistency, made concrete.

The write to the entity is committed the moment it returns. The view catches up milliseconds later.

t0POST /customers
t0+2msEntity persists customerCreated
t0+3ms200 returned to caller
t0+5msView updater sees event
t0+7msRow appears in customers_table
t0+8msGET /customers returns it
For read-your-write, hit the entity directly (by ID). Views are for list/search.
Endpoints

HTTP endpoints are the only external door by default.

Entities and views are not directly reachable from the network. Endpoints are the boundary.

@HttpEndpoint

Class-level annotation with a base path.

@Get / @Post / @Put / @Delete

Method-level annotations that map to routes.

@Acl

Access control — who can hit this endpoint.

Endpoints are stateless. All state lives in entities and views. Endpoints just translate HTTP ↔ domain calls.
Endpoints

CustomerEndpoint walkthrough.

api/CustomerEndpoint.java
@HttpEndpoint("/api")
@Acl(allow = @Acl.Matcher(principal = Acl.Principal.INTERNET))
public class CustomerEndpoint {

  private final ComponentClient client;

  public CustomerEndpoint(ComponentClient client) {
    this.client = client;
  }

  @Post("/customers")
  public CreateCustomerResponse create(CreateCustomerRequest req) {
    var cmd = new CreateCustomer(req.firstName, req.lastName);
    client.forEventSourcedEntity(req.customerId)
        .method(CustomerEntity::create)
        .invoke(cmd);
    return new CreateCustomerResponse(
        req.customerId, req.firstName, req.lastName);
  }

  @Get("/customers")
  public customerEntry list() {
    return client.forView().method(CustomerView::getAll).invoke();
  }
}
  • ComponentClient is how endpoints call components
  • Injected by the runtime — no DI setup
  • forEventSourcedEntity(id) targets one entity instance
  • forView() queries a view
  • .method(…::…) is a method reference to the handler
  • Request & response records are plain Java
Endpoints

How this differs from a Spring controller.

Spring MVC / Boot
  • @RestController → @Service → @Repository → DB
  • Transactions span layers; state is in the DB
  • Concurrency is your problem
  • Distribution is your problem
Akka SDK
  • Endpoint → ComponentClient → entity or view
  • State lives in the entity; the runtime persists it
  • Single-writer per entity ID — no locking
  • Sharding, replication, retries: runtime concerns
Same three-layer feel. Different guarantees underneath.
Wiring

One POST, end to end.

Every hop is in one JVM at dev time, and distributed transparently in prod.

1 · HTTPPOST /api/customers arrives
2 · EndpointCustomerEndpoint.create() runs
3 · ClientComponentClient targets customer-42
4 · EntityCustomerEntity.create() validates, persists event
5 · ViewCustomersUpdater sees customerCreated, writes row
6 · GETGET /api/customers returns the row
Steps 4 and 5 happen in parallel from the caller's perspective — the 200 returns after step 4.
Testing

TestKit — unit tests for stateful things.

No cluster, no HTTP. In-JVM, fast, deterministic.

CustomerEntityTest.java
var testKit = EventSourcedTestKit.of(CustomerEntity::new);

var result = testKit.method(CustomerEntity::create)
    .invoke(new CreateCustomer("Ada", "Lovelace"));

assertThat(result.getReply()).isEqualTo("customer-1");
assertThat(result.getNextEventOfType(customerCreated.class))
    .extracting(customerCreated::firstName)
    .isEqualTo("Ada");
  • Instantiates the entity in memory
  • Feeds it commands; asserts on emitted events and reply
  • Chains multiple commands to test evolution
  • No mocking — the real entity, minus the cluster
  • Views & endpoints have their own testkits too
