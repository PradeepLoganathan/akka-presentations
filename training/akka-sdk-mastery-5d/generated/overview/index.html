Why agents belong here

An agent is a stateful, addressable, reactive LLM caller.

That's a mouthful. Look at each word — you already have infrastructure for it.

Stateful

Session memory persists across turns. Same runtime the entities use.

Addressable

Session id routes you to the same agent instance. Same sharding as entities.

Reactive

Agents can be called from workflows, consumers, endpoints — anywhere a component can.

You don't bolt an LLM onto Akka. You host it as a component and get the whole runtime for free.
Agents

An Akka agent, anatomically.

Minimum viable agent
@ComponentId("support-agent")
public class SupportAgent extends Agent {

  public Effect<String> ask(String question) {
    return effects()
        .systemMessage("You are a helpful assistant.")
        .userMessage(question)
        .thenReply();
  }
}
  • extends Agent — component base class
  • @ComponentId gives it identity for storage & routing
  • effects() builds an LLM interaction description
  • systemMessage / userMessage mirror the OpenAI chat contract
  • .thenReply() hands the model's answer back to the caller
No SDK setup, no HTTP wiring, no client library. The runtime handles model calls the same way it handles persistence.
Agents

What actually happens when you .thenReply()?

The runtime turns your effect into a bounded loop with tool calls, memory reads, and error handling.

1Load session memory (if enabled)
2Build chat: system + memory + user
3Call the model provider (OpenAI)
4If tool calls: execute, feed results, repeat
5Persist new turn to memory
6Return the final response
Step 4 loops up to max-tool-call-steps (default 100). Beyond that the runtime stops even if the model wants more.
Agents

Model providers — first-class in the SDK.

ProviderWhere it fits
OpenAIDefault in the sample. Strong tool-use, wide model choice, well-known API.
AnthropicClaude family. Excellent for long-context and instruction-following.
Google AI GeminiMultimodal (image / audio) if you need it.
HuggingFaceAny hosted model with an HF endpoint.
Ollama / LocalAIAir-gapped or dev-loop with a local model. No cloud API key.
Different agents in the same service can use different providers. Config-driven — no code change to switch models.
Agents

All in application.conf.

src/main/resources/application.conf
akka.javasdk {
  agent {
    # Which provider agents default to
    model-provider = "akka.javasdk.agent.openai"

    openai {
      model-name  = "gpt-4o-mini"
      temperature = 0.2
      # api-key = ${?OPENAI_API_KEY}   (inherited default)
    }
  }
}
  • SDK ships default openai/anthropic/gemini configs
  • Override only what you need — model name, temperature, base URL
  • api-key = ${?OPENAI_API_KEY} is already in the reference config
  • Fail-fast on startup if the key is missing
  • Individual agents can override via .model(ModelProvider.openAi(...))

Agents

The Effect.Builder, one call at a time.

SupportAgent.ask()
public Effect<String> ask(String question) {
  return effects()
      .memory(MemoryProvider.limitedWindow().readLast(10))
      .systemMessage(SYSTEM_PROMPT)
      .tools(this)              // scan @FunctionTool methods on this class
      .userMessage(question)
      .thenReply();
}
  • .memory(...) — MemoryProvider (LimitedWindow / Disabled / Custom)
  • .systemMessage(...) — guardrails, role, constraints
  • .tools(this, ...) — objects or classes hosting @FunctionTool methods
  • .userMessage(...) — the actual question
  • .thenReply() — return the model's text response
Agents

The system message is the contract.

SupportAgent.SYSTEM_PROMPT

assistant for the sample e-commerce service in the
akka-workshop repository.

Rules:
- Never invent customer ids or order ids. Use the values
  the user provides, verbatim.

Agents

@FunctionTool — a method the model can call.

Anatomy of a tool method
@FunctionTool(description =
    "Look up a customer's profile by customerId. " +
    "Returns first name, last name, deleted flag.")
public String getCustomer(
    @Description("The customer id, e.g. cust-001")
    String customerId) {
  var s = client.forEventSourcedEntity(customerId)
      .method(CustomerEntity::get).invoke();
  if (s.customerId() == null)
    return "No customer found for " + customerId;
  return String.format(
      "customerId=%s firstName=%s deleted=%s", ...);
}
  • description is what the LLM reads to pick a tool
  • @Description on params helps the LLM fill them right
  • Return String — that's what the model sees
  • Format the return so the model can quote it later
  • "No data" is a valid answer; never fabricate

SupportAgent's four tools.

Each one hits a component we built on previous days. The LLM chooses which to call.

ToolWhat it queries

Agents

Memory: three built-in strategies.

ProviderBehaviour
DisabledNo history. Each turn is a cold start.
LimitedWindowBounded history — last N messages or K bytes. Sample uses this.
FromConfigReads memory settings from application.conf. Central overrides.
CustomPlug in your own SessionMemory implementation. Redis, external store, whatever.
Memory is persisted by the runtime — the same cluster mechanism as entities. No external cache needed.
Agents

LimitedWindow: bounded and observable.

Common shapes
// last N messages, read and write
MemoryProvider.limitedWindow().readLast(10);

// bounded by byte size (reference.conf default)
// akka.javasdk.agent.memory.limited-window.max-size = 510 KiB

// read only (agent sees history, doesn't record its turn)
MemoryProvider.limitedWindow().readOnly();

// write only (record but don't feed back)
MemoryProvider.limitedWindow().writeOnly();

// turn it off entirely
MemoryProvider.none();
  • FIFO eviction — oldest turn drops first
  • 510 KiB default max — safe for cluster routing
  • readOnly for a passive observer agent
  • writeOnly for a "log-only" mode during rollout
  • Memory is per session id, shared across agents in the same session
Agents

Session id is the router key.

Same session id → same memory. Different sessions → isolated conversations.

SupportEndpoint
@Post("/support/{sessionId}")
public Answer ask(String sessionId, Question req) {
  String answer = client.forAgent()
      .inSession(sessionId)
      .method(SupportAgent::ask)
      .invoke(req.question());
  return new Answer(sessionId, answer);
}
  • Caller chooses the session id — per-user, per-ticket, per-thread
  • forAgent().inSession(id) is how ComponentClient routes
  • Memory is scoped to the session — not the agent
  • Two different agents in the same session share memory
  • Sessions never explicitly end — memory bounds them naturally
Agents

Structured responses: responseAs & map.

Parsing model output
public record Summary(String status, String nextAction) {}

public Effect<Summary> summarise(String orderId) {
  return effects()
      .systemMessage("Return JSON matching the Summary schema.")
      .tools(this)
      .userMessage("Summarise order " + orderId)
      .responseAs(Summary.class)
      .thenReply();
}
  • .responseAs(Type.class) — deserialises JSON to a record
  • .map(fn) — transform the raw string
  • Use for tickets, decisions, workflow inputs
  • The runtime handles parse errors as JsonParsingException
  • Prompt the schema explicitly — the model can't infer field names
Agents

Failures the runtime catches for you.

Agent effects have a first-class error taxonomy. Recover per case, don't blanket-catch.

ExceptionWhen
RateLimitExceptionProvider throttled you. Back off, retry, or degrade.
ModelTimeoutExceptionModel didn't respond in time. Consider a shorter prompt.
ToolCallLimitReachedExceptionModel tried to call tools too many times. Simplify or add a tool.
ToolCallExecutionExceptionYour tool threw. Fix the tool or return a plain error string.
JsonParsingExceptionStructured response didn't parse. Tighten the schema prompt.
InternalServerExceptionProvider 5xx. Retry with backoff.
Use .onFailure(Function<Throwable, String>) to substitute a graceful response per exception type.
Agents

Multi-agent systems, without a framework.

Multiple agents in the same service, same runtime. Sharing sessions if they should, isolated if they shouldn't.

Same session

Agents share memory. A "triage" agent hands off to a "billing" agent, both see the transcript.

Different sessions

Isolated. Useful for tenant separation or batch workers.

AgentRegistry

@AgentDescription makes agents discoverable so a supervisor agent can list & delegate.

Multi-agent lives in the same Java service — no bus, no coordinator, no separate deployment.
MCP

MCP: tools your agents didn't write.

Model Context Protocol lets external LLM clients (Claude Desktop, Cursor, other agents) call tools you expose over HTTP.

You expose

Tools, resources, prompts — via annotations on plain Java methods.

Clients call

JSON-RPC over HTTP. Streamable. Discoverable via tools/list.

Auth

Same @Acl annotations as any endpoint. Same identity model.

Two audiences for the same domain: your internal agent uses @FunctionTool; external clients use @McpTool.
MCP

Testing

The pyramid, bent for stateful systems.

The classic unit → integration → e2e shape still applies, but each layer means something different when your components hold state.

LayerWhat to test
Unit (TestKit)One component in isolation. State transitions, event emissions, tool calls.
IntegrationMultiple components in the same JVM. Workflow + entity + view together.
ContractEvent schema evolution. Old events replayed by new code.
ChaosRuntime failures. Kill the JVM mid-workflow; assert recovery.
LoadThroughput at target scale. Entity sharding under pressure.
If your tests all live at the top, you're testing HTTP shape. If they all live at the bottom, you're testing nothing that matters at 2am.
Testing

WorkflowTestKit — step by step.

OrderWorkflowTest.java (excerpt)
var testKit = WorkflowTestKit.of(OrderWorkflow::new);

var started = testKit.method(OrderWorkflow::start)
    .invoke(new StartOrder("c1",
        List.of(new OrderLine("sku-1", 1, 100))));

assertThat(started.getReply()).startsWith("order-");
assertThat(testKit.currentState().totalCents()).isEqualTo(100L);

// advance one step at a time
var t1 = testKit.getTransition();
assertThat(t1.stepName()).isEqualTo("reserve");

testKit.runStep("reserve");
var t2 = testKit.getTransition();
assertThat(t2.stepName()).isEqualTo("charge");
  • In-JVM, no cluster, no HTTP
  • Mock component clients per step for isolation
  • Assert transitions, state, scheduled timers
  • Run one step at a time or the whole workflow
  • Time is virtual — test 24h timers in milliseconds
Testing

EventSourcedTestKit, for the Order entity.

OrderEntityTest.java (shape)
var tk = EventSourcedTestKit.of(OrderEntity::new);

// given: an order was placed
tk.method(OrderEntity::place)
  .invoke(new PlaceOrder("c1", lines));

// when: mark reserved
var r = tk.method(OrderEntity::markReserved)

// then

  • Fastest possible feedback for domain logic
  • Test replay: tk.replayEvents(...) and assert state

  • Snapshot a scenario with builders (see slide 36)

CustomerAuditConsumerTest.java (shape)
var tk = ConsumerTestKit.of(CustomerAuditConsumer::new);

var result = tk.onEvent(

// side effects captured for assertion

Agent testing: mock the model.

SupportAgentTest.java (shape)
var tk = AgentTestKit.of(SupportAgent::new)
    .withMockedModel("gpt-4o-mini", scriptedResponses);

Testing

Integration: real components, in one JVM.

TestKit supports full multi-component scenarios — entity + workflow + consumer + view — without leaving the JVM.

TestKitSupport

Boots the runtime in-process. Same behaviour as prod, minus the network.

Scenario tests

POST an order, wait for terminal status, assert view + audit log content.

Awaitility

Eventually-consistent assertions with bounded polling. No sleep(2000) hacks.

One integration test per user-visible outcome. Ten unit tests per component. That's the ratio.
Testing

Chaos: assert what survives.

Not "does it work?" but "what happens when it doesn't?" Test the runtime's failure semantics on your workflow.

Mid-step kill

POST an order, watch it get to charge, kill the JVM, restart, assert SHIPPED.

Consumer redelivery

Kill mid-consume; assert the same event is redelivered on restart; assert idempotency.

Provider outage

Point OpenAI base URL at a broken host; assert onFailure handler engages, workflow doesn't corrupt.

Chaos tests are expensive to run and priceless when a production incident matches the scenario.
Testing

Contract tests: events are forever.

An event journaled today gets replayed by code you'll write in 2028. Break the schema, break the past.

Schema tests

Fixture files of historical event JSON. Assert the new record types can deserialize each one.

Replay tests

Given a corpus of events, assert the current state matches the expected snapshot.

Migration tests

When you must break, migrate: old event → new event via a translator. Test both sides.

Rule: adding a field is safe (with a default). Renaming, removing, changing type — assume it will break replay, and test accordingly.
Testing

Load: what does sharding look like at 10k RPS?

Load tests answer questions unit tests can't: throughput at your scale, tail latency, entity distribution, LLM token burn.

Tooling

k6, Gatling, or plain Go with worker pools. Whatever your team already runs.

Scenarios

Not "1 endpoint, max RPS." "1000 concurrent workflows across 100 entities."

Success signals

P99 latency, workflow completion time, view lag. Not just RPS.

If the load test doesn't distribute across entity ids, you're testing one shard — not the system.
Testing

Idempotency: the non-negotiable.

At-least-once delivery means "duplicates happen." Every command handler, every consumer, every timer target needs a test that says: "the second time is a no-op."

Entity commands

Send the same command twice. Assert the second attempt is rejected or idempotent.

Consumer handlers

Feed the same event twice. Assert the downstream state is identical to feeding it once.

Timer targets

Test the command the timer calls, not the timer. Idempotence is the target's problem.

A test that only fires the command once tells you nothing about production. Fire it three times.
Testing

Test data: builders, not fixtures.

Ten integration tests that all take a full-order fixture become ten broken tests when you rename a field. Build the shape you need per test.

Fluent builders

OrderFixture.placed().withLine("sku-1", 2).build() reads as intent.

Property-based

Generate random valid inputs. Catch edge cases you'd never write.

Golden files

Only for output-shape assertions. Not for input.

If two tests differ only in one field, the fixture is the wrong shape. Make it a parameter.
Testing

What not to test.

Time and complexity are finite. Spend them on what changes when your code changes.

Don't test the runtime

"Does Akka persist events?" — that's their test, not yours.

Don't test framework glue

Testing @ComponentClient wiring proves nothing. Test the behaviour it enables.

Don't test the LLM

You can't. Test your prompt, your tools, your guardrails.

Rule of thumb: if the test would still pass after removing your code, you're testing the wrong thing.
Testing

Staging is not a test environment.

It's a rehearsal. What runs in staging should be identical to production — same runtime, same config, real data shapes.

CI tests

Everything in the pyramid up through integration. No network egress. Deterministic.

Staging

Full deploy. Real LLM keys (with lower quotas). Smoke tests. Chaos exercises on schedule.

Production canary

Fraction of traffic to the new version. Real monitoring. Real rollback path.

If staging finds bugs that CI didn't, add the missing CI test. Staging is the last check, not the first.

21 new instance started, health-checked
3Traffic shifted, entities rebalance to new instance
41 old instance drained + terminated
5Repeat until all N instances are new
6In-flight workflows resume on new instances via replay
Requires backward-compatible event schemas. See contract tests (slide 33).
Deploy

Autoscale: request rate + entity distribution.

Scale on what actually matters for stateful workloads: not just CPU, but shard load and workflow queue depth.

Signals

Request rate, active workflows, entity shard imbalance, LLM queue depth.

Bounds

Min instances (never scale to zero for stateful services). Max, based on cost budget.

Cooldown

Scale-up fast, scale-down slow. Entity migration has a real cost.

Scale-to-zero on a stateful service means cold-start rehydration and lost session memory. Don't.
Multi-region

Why go multi-region?

Three reasons. Different from each other. Each drives a different topology.

Latency

Users on three continents. Serving them from one region hurts them.

Availability

Regional outages are real. You need to survive AWS-us-east-1 going down.

Sovereignty

EU customer data stays in EU regions. Regulatory. Non-negotiable.

Different reasons; different topologies. Don't build for all three when you only need one.
Multi-region

Two shapes: active/active vs primary/replica.

Active/active
  • Every region writes and reads locally
  • Entities replicated across regions
  • Local reads always; local writes when the entity is "home" here
  • Best for latency + availability
  • Conflict resolution matters (usually last-write-wins per entity)
Primary/replica
  • One region owns writes for a slice of entities
  • Replicas serve local reads only
  • Failover on primary loss = promotion
  • Best for sovereignty + simpler consistency
  • Slightly higher write latency for far-away callers
Multi-region

Config-first: no code changes.

Terminal
# create a project spanning three regions
akka projects new akka-workshop-prod \
    --region gcp-us-east1 \
    --region gcp-europe-west1 \
    --region gcp-asia-southeast1

# set primary region for a service
akka services deploy akka-workshop \
    ghcr.io/pradeepl/akka-workshop:1.0 \
    --region gcp-us-east1

# add a replica region
akka services replicate akka-workshop \
    --to gcp-europe-west1
  • Regions are project-level; services opt in
  • Sample code compiles unchanged — the runtime handles replication
  • Entity id determines which region "owns" it (consistent hashing)
  • Views project locally per region
  • Workflows resume in the region where they started
Multi-region

Entity semantics across regions.

The mental model shifts slightly. Local reads always. Writes go to the entity's home region.

Reads

Always local. Views project per region, entity state is replicated with bounded staleness.

Writes

Routed to the entity's home region — usually one round-trip more if the caller is elsewhere.

Consistency

Single-writer per entity globally. No conflicting writes at the entity level.

CustomerEntity for customer-42 has one home region. Reads happen anywhere. Writes go to customer-42's region.
Multi-region

Workflows & timers across regions.

Workflows are like entities: one home region per instance. Timers persist with the workflow.

Workflow home

Determined by the workflowId. OrderWorkflow for order-abc runs in one region for its whole life.

Timers

Fire in the workflow's home region. Not affected by request routing.

Cross-region calls

Workflow calls entities in their home regions. Latency-aware; SDK handles it.

Design tip: keep tightly-coupled aggregates (Order + its Customer) in the same region if you can — the SDK does this by default when they share an id prefix.
Multi-region

When a region disappears.

Failover isn't a manual playbook — it's part of the runtime. Your job is knowing what to expect.

1Region A becomes unreachable
2Traffic reroutes to nearest healthy region
3Entities homed in A are promoted in surviving regions
4Workflows resume from last checkpoint elsewhere
5Bounded window of stale reads until replication catches up
6Region A returns → replicas resync, ownership rebalances
Test it: akka regions drain gcp-us-east1 in staging. Watch what actually happens.

Runbooks

Runbooks: what a tired oncall needs.

Not "here's how the system works." That's docs. A runbook answers "the alert fired — what do I do right now?"

The trigger

Exact alert name and severity. Not paraphrased.

Verify

One command to confirm the alert is real. Screenshots of what the console should show.

Mitigate

Numbered steps. Bounded. Reversible if possible. Escalation trigger if none work.

Rule: if the runbook says "investigate" without a next step, it's not a runbook. It's an aspiration.
Runbook example

Alert: OrderWorkflow stuck > 5min.

Steps
1. Verify:
   akka logs --grep "workflow" --tail 100
   Open console: which workflow instance? Which step?

2. Diagnose:
   akka services describe akka-workshop
   Look for InventoryEntity or PaymentEntity errors.

3. Common causes:
   - Payment provider outage (check status.stripe.com)
   - Inventory reserved-orders leak (see runbook 61)
   - Model provider timeout (check /metrics)

4. Mitigate:
   - Transient: wait for provider recovery
   - Persistent: transition workflow to compensate:
     akka services command akka-workshop \
       order-workflow <id> forceCompensate

5. Escalate if:
   > 20 workflows stuck OR provider outage > 15min
  • Specific alert name, no ambiguity
  • Verify step confirms it's real
  • Bounded diagnostic checks
  • Concrete mitigation commands
  • Clear escalation trigger
  • Living doc — update after every incident
Runbook example

Alert: Entity shard unavailable.

Steps
1. Verify:
   akka services status akka-workshop
   Look for READY count < expected instances.

2. Diagnose:
   Which shards are affected? Which entity ids fall in them?
   akka logs --grep "shard" --level warn --tail 200

3. Common causes:
   - Rolling deploy in progress (transient, wait it out)
   - Region-level networking issue (check status page)
   - Persistence backend hiccup

4. Mitigate:
   - If rolling deploy: monitor, do nothing, alert stops
   - If region issue: multi-region traffic will reroute
   - If backend: escalate to platform team

5. What NOT to do:
   Do not force-restart the service. Sharding recovers on its own.
  • Explicit "do not" section — wrong instincts documented
  • Distinguish transient from persistent
  • Multi-region behaviour called out (autonomic)
  • Escalation path clear
Runbook example

Alert: LLM error rate > 5%.

Steps
1. Verify:
   akka logs --grep "RateLimit\|ModelTimeout" --tail 50
   Check exception classes; each has a different mitigation.

2. Diagnose:
   RateLimitException  → quota exhausted or throttled
   ModelTimeoutException → provider slow or overloaded
   InternalServerException → provider 5xx
   ToolCallLimitReached → a bad system prompt is looping

3. Mitigate (short-term):
   - Rate-limit: enable backoff, drop non-critical requests
   - Timeout: downshift to gpt-4o-mini if not already
   - Provider 5xx: fail-open with cached last-known answer
   - Tool loop: check git log for recent prompt changes

4. Mitigate (long-term):
   - Cost: pre-cache tool outputs; batch requests
   - Reliability: multi-provider fallback (Anthropic secondary)

5. Escalate if:
   > 30min sustained, business hours, customer-facing.
  • Distinct handling per exception type
  • Short-term (mitigate) vs long-term (fix)
  • Fail-open policy called out explicitly
  • Multi-provider fallback as an option
Ops

Capacity planning: what to size for.

Stateful workloads size on different dimensions than stateless ones. Instance count is only one lever.

Active entities

Entities in memory per node. Passivation reclaims idle ones.

Workflow throughput

How many workflows can transition per second? Set by step call latency, not CPU.

View lag

How stale can reads be? Sets consumer parallelism.

LLM tokens

Per-agent-call. Budget = model choice × memory window × tool calls.

Storage

Journal grows forever unless you archive. Design event schema for retention early.

Egress

LLM calls are egress. Multi-region calls are egress. Both add up.

Ops

Cost: where the money actually goes.

For a typical Akka service with agents, LLM tokens dominate. Storage is a distant second. Compute is a rounding error.

LLM tokens

Model choice matters — gpt-4o-mini vs gpt-4o is ~15x. Memory window doubles cost linearly.

Storage

Event journal + snapshots. Grows forever unless you archive cold data.

Compute

Managed runtime is priced per instance-hour. Small compared to LLM in most agentic services.

Guardrail: budget alarms per provider. If someone accidentally deploys gpt-4o to prod and forgets, you find out in an hour, not in the finance meeting.
