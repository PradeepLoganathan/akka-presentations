Testing

The pyramid, bent for stateful systems.

The classic unit → integration → e2e shape still applies, but each layer means something different when your components hold state.

LayerWhat to test
Unit (TestKit)One component in isolation. State transitions, event emissions, tool calls.
IntegrationMultiple components in the same JVM. Workflow + entity + view together.
ContractEvent schema evolution. Old events replayed by new code.
ChaosRuntime failures. Kill the JVM mid-workflow; assert recovery.
LoadThroughput at target scale. Entity sharding under pressure.
If your tests all live at the top, you're testing HTTP shape. If they all live at the bottom, you're testing nothing that matters at 2am.
