Motivation

"Place an order" isn't one thing.

It's at least three: reserve inventory, charge payment, mark shipped. Any of them can fail. All of them can succeed on retry.

Reserve

Inventory service must set aside the SKUs before we charge.

Charge

Payment gateway takes the money. Might decline. Might time out.

Ship

Fulfilment gets the order. Depends on both of the above.

If step 2 fails after step 1 succeeded, we need to release the inventory. That's the whole game.
Why not just…

Chain entity calls from the endpoint?

Naive endpoint (don't do this)
@Post("/orders")
public String place(PlaceOrder req) {
  // step 1
  inventory.reserve(req);
  // step 2 — if this throws...
  payment.charge(req);
  // ...step 3 never runs and inventory is stuck reserved
  fulfillment.ship(req);
  return "OK";
}
  • Not durable — if the endpoint process dies mid-flight, work is lost
  • No compensation — nothing rolls back the inventory reservation
  • No retry — caller has to know to retry, and safely
  • State lives in the caller's head, not on disk
  • You'll rebuild half a workflow engine before the demo
The alternative

Workflows: durable orchestrators.

A workflow is a component that models a multi-step process. State is persisted, steps are journaled, transitions are explicit.

Durable

State survives crashes. On restart, the workflow resumes from its last step.

Explicit

Steps are named. Transitions are decisions, not implicit control flow.

Compensable

Failed step? Transition to a compensate step that undoes the earlier work.

Think "state machine + persistence + retry + observability," not "async/await."
In the family of patterns

Workflow vs. saga vs. choreography.

PatternHow steps coordinate
Orchestration
(Akka workflow)Central coordinator drives each step. Explicit compensation. Easy to reason about.
ChoreographyEach service reacts to events. No coordinator. Emergent flow. Great for loose coupling.
SagaUmbrella term for either. Focus is on compensation across distributed steps.
Step functions (cloud)Like orchestration, but the coordinator is proprietary infra.
Workflows here = orchestration, in-process, portable, testable.
The workflow needs a subject

Introducing the Order aggregate.

Second event-sourced entity in the sample. Six events cover the full lifecycle: placed, reserved, paid, shipped, failed, cancelled.

domain/events/OrderEvents.java
public sealed interface OrderEvents {
  record orderPlaced(...)         implements OrderEvents {}
  record inventoryReserved(...)   implements OrderEvents {}
  record paymentRecorded(...)     implements OrderEvents {}
  record orderShipped(...)         implements OrderEvents {}
  record orderFailed(...)          implements OrderEvents {}
  record orderCancelled(...)       implements OrderEvents {}
}
  • Each event = a step in the lifecycle
  • Terminal states: SHIPPED, FAILED, CANCELLED
  • Sealed = compiler-checked reducer
  • OrderEntity has command handlers to persist each event
  • The workflow decides which events fire, in what order
OrderEntity

Same event-sourced pattern, richer statuses.

domain/entities/OrderEntity.java (excerpt)
@ComponentId("Order")
public class OrderEntity
    extends EventSourcedEntity<OrderState, OrderEvents> {

  public Effect<String> markReserved(MarkReserved cmd) {
    if (currentState().status() != OrderStatus.PLACED)
      return effects().error(
          "Cannot reserve from status " + currentState().status());
    return effects()
        .persist(new inventoryReserved(currentState().orderId()))
        .thenReply(__ -> "OK");
  }
}
  • Command handlers enforce the state machine at the entity level
  • Illegal transitions rejected as errors, not silently ignored
  • The workflow drives the calls but the entity guards its own invariants
  • Two layers of safety: workflow logic + entity validation
  • Even a buggy workflow can't put the order in an invalid state
OrderWorkflow

A workflow, at first glance.

domain/workflows/OrderWorkflow.java (top)
@ComponentId("order-workflow")
public class OrderWorkflow
    extends Workflow<OrderWorkflow.State> {

  public record State(
      String orderId, String customerId,
      List<OrderLine> lines, long totalCents) {}

  private final ComponentClient client;

  public OrderWorkflow(ComponentClient client) {
    this.client = client;
  }

  @Override
  public WorkflowDef<State> definition() {
    return workflow()
        .addStep(reserveStep())
        .addStep(chargeStep())
        .addStep(shipStep())
        .addStep(compensateStep());
  }
}
  • @ComponentId like any component
  • Workflow<State> — typed on the workflow's own state
  • Constructor accepts ComponentClient to call entities
  • definition() lists the steps in scope
  • Method references keep the code shape narrow
Two states, distinguished

Workflow state is not entity state.

The workflow tracks orchestration context. The Order entity tracks domain truth. Same orderId, different concerns.

OrderWorkflow.State
  • What is this workflow instance working on?
  • orderId + customerId + lines + total
  • Persisted per workflow instance
  • Lost when workflow ends or is deleted
OrderState (entity)
  • What is the truth of this order?
  • orderId + lines + total + status
  • Persisted as an event log; never deleted
  • Answers "what happened to this order?"
Workflow state = what the orchestrator needs to keep going. Entity state = what history demands to be true.
A step, dissected

Every step is call → andThen.

reserveStep()
private Step reserveStep() {
  return step("reserve")
      .call(() ->
          client.forKeyValueEntity(INVENTORY_ID)
              .method(InventoryEntity::reserve)
              .invoke(new ReserveInventory(
                  currentState().orderId(),
                  currentState().lines())))
      .andThen(String.class, result -> {
        if ("OK".equals(result)) {
          client.forEventSourcedEntity(currentState().orderId())
              .method(OrderEntity::markReserved)
              .invoke(new MarkReserved());
          return effects().transitionTo("charge");
        }
        return effects().transitionTo(
            "compensate", "inventory:" + result);
      });
}
  • step("name") — unique, referenced in transitions
  • .call(Supplier) — the action the step performs
  • .andThen(Class, Function) — decide the next step from the call's result
  • transitionTo("name") — where to go next
  • Call retries on failure; andThen only runs on success
The superpower

Workflows call other components.

Entities can't call other entities. Workflows can. That's the point — workflows are where cross-aggregate coordination lives.

Workflow → KVE

InventoryEntity (KVE) reserve. Idempotent by orderId.

Workflow → KVE

PaymentEntity (KVE) charge. Returns OK or DECLINED.

Workflow → ES entity

OrderEntity markReserved / markPaid / markShipped. Records the fact.

Rule of thumb: if a step needs to talk to two aggregates, that step belongs in a workflow.
The saga move

Compensation is just another step.

compensateStep()
private Step compensateStep() {
  return step("compensate")
      .call(String.class, reason -> {
        client.forKeyValueEntity(INVENTORY_ID)
            .method(InventoryEntity::release)
            .invoke(new ReleaseInventory(currentState().orderId()));

        client.forEventSourcedEntity(currentState().orderId())
            .method(OrderEntity::markFailed)
            .invoke(new MarkFailed(reason));

        timers().delete(ABANDON_TIMER_PREFIX + currentState().orderId());
        return reason;
      })
      .andThen(String.class, __ -> effects().end());
}
  • Any step can transition here with a reason string
  • Releases the inventory reservation (the earlier work)
  • Marks the order failed — entity records the fact
  • Cancels the abandon timer — no ghost cancellations later
  • Ends the workflow — no more transitions
The durability guarantee

Kill the JVM. Then start it again.

Workflow state is persisted after every transition. On restart, the runtime looks up the last known step and resumes from there.

1start() persists State, transitions to "reserve"
2reserveStep calls Inventory, succeeds
3andThen writes "transition to charge"
4JVM crashes here
5JVM starts; runtime sees last step = charge
6chargeStep runs. Order completes.
You get this for free. The runtime does the checkpointing. You write the steps.
Advanced: retries per step

RecoverStrategy: retries, then failover.

definition() with recovery
@Override
public WorkflowDef<State> definition() {
  var retryThenCompensate = RecoverStrategy
      .maxRetries(3)
      .failoverTo("compensate");

  return workflow()
      .addStep(reserveStep(), retryThenCompensate)
      .addStep(chargeStep(),  retryThenCompensate)
      .addStep(shipStep())
      .addStep(compensateStep());
}
  • Default: retry with exponential backoff, forever
  • maxRetries(n) — give up after n
  • .failoverTo("stepName") — on give-up, jump here
  • Applied per step via addStep(step, strategy)
  • Combine with a compensate step for saga-style rollback
Advanced: whole-workflow failover

When the entire workflow gives up.

Workflow-level failover
return workflow()
    .addStep(reserveStep())
    .addStep(chargeStep())
    .addStep(shipStep())
    .addStep(compensateStep())
    .timeout(Duration.ofMinutes(30))
    .failoverTo("compensate",
                "workflow timeout",
                MaxRetries.of(0));
  • Applies when the whole workflow exceeds its budget
  • Runtime transitions to the failover step with the given input
  • You typically point this at your compensate step
  • Different from step-level recovery: catches "we're just stuck"
  • maxRetries here bounds retries of the failover step
Advanced: budgets

Two clocks: step and workflow.

Step timeouts guard against one call hanging. Workflow timeout guards against the whole business process stalling.

Step timeout
  • Applies to a single step's .call() action
  • Set via step(...).timeout(Duration.ofSeconds(10))
  • Or a workflow-wide default via defaultStepTimeout()
  • Hitting it triggers the step's RecoverStrategy
Workflow timeout
  • Applies to the whole workflow's lifetime
  • Set via workflow().timeout(Duration.ofMinutes(30))
  • Hitting it triggers workflow-level failover (or ends)
  • Use for "abandoned checkout after N minutes"
Advanced: finishing

end() vs delete().

Both terminate the workflow. They differ in what remains queryable and how downstream consumers observe closure.

effects().end()
  • Workflow stops. State stays.
  • GET the workflow still returns its final state
  • No more transitions accepted
  • Default choice for "finished successfully"
effects().delete()
  • Workflow stops. State scheduled for deletion.
  • Delay is configurable so downstream consumers can observe the deletion
  • After the delay, the workflow instance is gone
  • Use for "one-shot, no need to look up later"

Timers

Send a command to your future self.

Timers schedule a component call for later — a durable "check back on this in 30 minutes."

Durable

Persisted. Fires even if the JVM restarts before the delay elapses.

At-least-once

Retries on transient failure. Your handler should be idempotent.

Named

Reuse the name to overwrite an existing timer. Use it to delete/cancel.

Uses: abandoned-cart cancels, session TTLs, "check in after payment provider webhook."
Timers

TimerScheduler in three calls.

The API
// schedule
timers().createSingleTimer(
    "abandon-order-42",
    Duration.ofSeconds(30),
    client.forEventSourcedEntity("order-42")
        .method(OrderEntity::cancelIfNotPaid)
        .deferred(new CancelIfNotPaid())
);

// cancel
timers().delete("abandon-order-42");
  • timers() — available in workflows, entities, consumers
  • createSingleTimer(name, delay, deferredCall)
  • .deferred(cmd) — describes the future call but doesn't run it
  • Re-registering the same name replaces the pending timer
  • delete(name) cancels a pending timer
Timers

Where the abandon timer lives in Order.

OrderWorkflow.start() (excerpt)
public Effect<String> start(StartOrder cmd) {
  String orderId = commandContext().workflowId();
  ...

  timers().createSingleTimer(
      ABANDON_TIMER_PREFIX + orderId,
      ABANDON_TIMEOUT,   // Duration.ofSeconds(30) for demo
      client.forEventSourcedEntity(orderId)
          .method(OrderEntity::cancelIfNotPaid)
          .deferred(new CancelIfNotPaid()));

  return effects()
      .updateState(new State(...))
      .transitionTo("reserve")
      .thenReply(orderId);
}
  • Scheduled when the workflow starts
  • Deleted on success in chargeStep
  • Deleted on failure in compensateStep
  • Fires only if the order never reaches a terminal state
  • Entity's cancelIfNotPaid guards against races
Timers · war story

Timer ops belong in call, never in andThen.

A real bug we hit building this sample. Cost of learning: about 30 minutes of head-scratching.

Broken (chargeStep, first attempt)
.andThen(String.class, result -> {
  if ("OK".equals(result)) {
    client.forEventSourcedEntity(orderId)
        .method(OrderEntity::markPaid)
        .invoke(new MarkPaid());
    // throws IllegalStateException
    timers().delete(ABANDON_TIMER_PREFIX + orderId);
    return effects().transitionTo("ship");
  }
  ...
});
Fixed — move it into call()
.call(() -> {
  String result = client.forKeyValueEntity(orderId)
      .method(PaymentEntity::charge).invoke(...);
  if ("OK".equals(result)) {
    timers().delete(ABANDON_TIMER_PREFIX + orderId);
  }
  return result;
})
.andThen(String.class, result -> {
  ...
});
Rule: timer scheduling/cancellation is only valid inside a command handler or a step's call action. Not in andThen. Not in the reducer.
Timers

Testing time-based logic.

You don't want a 30-second sleep in your test suite. You also don't want to mock the whole scheduler.

Shrink the delay

Make ABANDON_TIMEOUT config-driven. Set 100ms in tests, 30s in demo, 24h in prod.

Assert the schedule

WorkflowTestKit exposes scheduled timers so you can assert "yes, a timer named X was set for Y."

Fire manually

Test the target command handler (cancelIfNotPaid) directly. The timer just calls it.

Consumers

Reactive processing of someone else's events.

A consumer subscribes to an entity's event stream. Each event fires a handler you write. Great for side effects that aren't projections.

Audit log

Persist every domain event to a searchable audit trail.

Notifications

Order shipped? Consumer emails the customer.

Integration

Republish events to Kafka, another Akka service, an external webhook.

Rule of thumb: if it's not a query projection, it's a consumer.
Consumers

Consumer or view? The common confusion.

View
PurposeServe queries
OutputA queryable table (rows keyed by entity)
Effect APIupdateRow(row)
Side effects?No — keep it pure
Consumer
PurposeReact to events with side effects
OutputWhatever the handler does — usually a call to another component

Consumers

Where events come from: @Consume.

Subscription options
@Consume.FromEventSourcedEntity(CustomerEntity.class)
public class CustomerAuditConsumer extends Consumer {
  // receives CustomerEvents
}

@Consume.FromKeyValueEntity(CustomerPreferencesEntity.class)
public class PreferencesConsumer extends Consumer {
  // receives state snapshots
}

public class ExternalConsumer extends Consumer {
  // receives messages from a topic
}
  • FromEventSourcedEntity — the event log
  • FromKeyValueEntity — state-change stream
  • FromTopic — external pub-sub
  • FromServiceStream — another Akka service's stream
  • One consumer = one subscription
Consumers

CustomerAuditConsumer, in full.

consumers/CustomerAuditConsumer.java
@ComponentId("customer-audit-consumer")
@Consume.FromEventSourcedEntity(CustomerEntity.class)
public class CustomerAuditConsumer extends Consumer {

  private final ComponentClient client;

  public CustomerAuditConsumer(ComponentClient client) {
    this.client = client;
  }

  public Effect onEvent(CustomerEvents event) {
    return switch (event) {
      case customerCreated e -> record(e.customerId(),
          "customer.created", e.FirstName() + " " + e.LastName());
      case customerDeleted e -> record(e.customerId(),
          "customer.deleted", e.FirstName() + " " + e.LastName());
    };
  }
}
  • One onEvent for the sealed interface; switch on subtype
  • Or one onEvent(subtype) per event type — both work
  • Handler calls the AuditLogEntity to append
  • Returns an Effect to say "processed"
  • Consumer is stateless — state lives in the entities it calls
Consumers

Three ways to finish handling.

Consumer.Effect builders
// standard: acknowledge the message
return effects().done();

// publish to a topic (if @Produce.ToTopic on the class)
return effects().produce(new AuditNotification(...));

// skip this one (do not retry)
return effects().ignore();
  • done() — success. Move to next event.
  • produce(msg) — publish a derived message. Also acknowledges.
  • ignore() — skip. Won't be retried.
  • Async variants exist for CompletionStage-returning code
  • Throwing an exception = retry (at-least-once)
Consumers

At-least-once means "sometimes twice."

Consumers can see the same event twice if a failure happens between processing and acknowledgement. Design for idempotency.

Idempotent keys

Use the event's unique offset or subject as part of the row/entity id.

Check before write

"If already appended, skip." Guard the target entity's command handler.

Idempotent side effects

PUTs > POSTs. Upserts > inserts. Named calls > anonymous.

If you can't make it idempotent, at least make the duplicate harmless. Log twice; don't email twice.
Consumers · sharp edge

Feed-style views need a topic in between.

A view that consumes an event-sourced entity's events collapses to one row per entity id. If you want one row per event, you need a hop.

1Entity emits events
2Consumer subscribes to events
3Consumer produce()s to a topic
4View subscribes to that topic
5View writes a row per message
6Query returns the feed
This is why the sample's AuditLogEntity holds the log itself — a feed view is out of scope for today.

Putting it together

What survives a kill -9?

The interesting question isn't "does it work?" It's "what happens when it doesn't?"

Workflow crash mid-step

Runtime restarts the step from the last committed transition. Idempotent step design pays off here.

Consumer crash mid-event

Event redelivered on restart. Idempotent target handler pays off.

Cluster node loss

Sharding rehomes entities and workflows to surviving nodes. Timers persist.

"Durable" is doing a lot of heavy lifting. The runtime is designed for these failures — not merely tolerant of them.
Putting it together

The saga pattern, in one screen.

You've now implemented one. Here's what to name and explain when someone asks.

Orchestrated saga

A workflow drives each step. Central place for decision logic.

Compensation, not rollback

You cannot undo a charged card. You issue a refund. Compensation is business-visible.

Idempotent participants

Each step's target must tolerate replay. Payment providers offer idempotency keys for this.

Two-phase commit doesn't survive real networks. Sagas do — because they were designed for failure.

WorkflowTestKit: walk the steps.

OrderWorkflowTest.java (excerpt)
var testKit = WorkflowTestKit.of(OrderWorkflow::new);

var started = testKit.method(OrderWorkflow::start)
    .invoke(new StartOrder("c1",
        List.of(new OrderLine("sku-1", 1, 100))));
